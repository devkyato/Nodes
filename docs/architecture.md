# Architecture

Flowchart Creator keeps the published runtime surface small: one HTML document, generated FlyonUI/Tailwind CSS, project-specific CSS, and one JavaScript module loaded as a classic script. FlyonUI and Tailwind are build-time development dependencies; the deployed editor does not contact a UI CDN or application server.

## State and history

The application state owns the document title, nodes, edges, selection, viewport, grid and snap preferences, mode, and snapshot history. Device-level preferences separately own reduced motion, automatic drafts, and connector defaults. Diagram-changing actions commit once at the end of an interaction, so pointer movement does not flood undo history.

## Coordinate system

Nodes and connectors use a fixed 2400 × 1600 logical workspace. The SVG viewport applies one pan-and-zoom transform. Pointer coordinates are converted back into logical coordinates before movement, resize, selection, or connector calculations. This keeps interaction correct at every zoom level.

## Rendering

Rendering is deterministic from state:

1. Connectors render beneath nodes.
2. Nodes render in `zIndex` order.
3. Selection outlines, handles, ports, and temporary gestures render in a separate SVG overlay.
4. Context actions, properties, settings, commands, and toasts render as accessible HTML controls above the canvas.

Each flowchart symbol is generated by the shared geometry renderer. Export reuses the same node and connector renderers, then omits the editing overlay and grid.

## Persistence

Projects serialize into versioned JSON or structured XML. Import normalizes missing style, lock, and grid fields against current defaults, allowing compatible older files to load. Manual checkpoints, automatic drafts, themes, and device preferences use separate namespaced local-storage keys.

## Interface system

`src/flyon.css` configures Tailwind CSS v4 and FlyonUI. The generated stylesheet supplies component primitives; `style.css` provides the compact Nodes identity, responsive editor layout, state-specific canvas styling, and reduced-motion overrides. Dropdowns, modal visibility, tabs, command search, and focus behavior are controlled locally so only one surface competes for attention at a time.

## Testing

The smoke suite starts the local server, opens an installed Chromium-family browser through the DevTools protocol, and verifies core workflows without a test framework. It covers FlyonUI surfaces, settings, commands, locking, themes, persistence, XML/JSON round-trips, and image export. CI runs the same suite before packaging and deployment.
